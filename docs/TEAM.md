## Entscheidungsprozess
- **PRs**: 1 Maintainer-Review reicht fuer Merge
- **Breaking Changes**: Issue oeffnen + 48h Diskussion
- **Releases**: Maintainer kann eigenstaendig taggen + publishen
- **Roadmap**: Im GitHub Project Board, Diskussion in GitHub Discussions

## Beitragen

## Release-Prozess
1. Conventional Commit auf feature-branch
2. PR → auto-label via semantic-release
3. 1 Maintainer approves
4. Squash-merge → CI/CD triggert automatischen Release

## Autonomie-Prinzipien
- Jedes Team arbeitet ohne zentrale Freigabe
- PRs koennen reviewt und gemergt werden
- Releases koennen eigenstaendig getaggt werden
- Issues koennen triagiert werden
- Eigene Roadmap-Entscheidungen
